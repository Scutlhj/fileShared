# 《重构》读书笔记

## 第 7 章 封装

分解模块时最重要的标准，就是识别出那些模块应该对外界隐藏的小秘密。数据结构是最常见的一种秘密。

### 7.1 封装记录

**动机**

记录型结构能直观地组织关联数据，但强迫区分"存储的数据"和"计算得到的数据"。对于可变数据，更偏爱使用类对象而非记录，因为类对象可以隐藏结构细节。如果数据不可变，可直接保存在记录里。


**范例**

```js
// 重构前
const organization = { name: "Acme Gooseberries", country: "GB" };

// 重构后
class Organization {
  constructor(data) {
    this._name = data.name;
    this._country = data.country;
  }
  get name() { return this._name; }
  set name(arg) { this._name = arg; }
  get country() { return this._country; }
  set country(arg) { this._country = arg; }
}
```

---

### 7.2 封装集合

**动机**

只对集合变量的访问进行封装，但依然让取值函数返回集合本身是不够的——集合成员变量可直接被修改，封装它的类则全然不知。应在类上提供"添加"和"移除"方法，使对集合的修改必须经过类。更好的做法是取值函数返回集合的副本，避免客户端的意外修改。

**范例**

```js
// 重构前
class Person {
  get courses() { return this._courses; }
  set courses(aList) { this._courses = aList; }
}

// 重构后
class Person {
  get courses() { return this._courses.slice(); }
  addCourse(aCourse) { this._courses.push(aCourse); }
  removeCourse(aCourse, fnIfAbsent = () => { throw new RangeError(); }) {
    const index = this._courses.indexOf(aCourse);
    if (index === -1) fnIfAbsent();
    else this._courses.splice(index, 1);
  }
}
```

### 7.3 以对象取代基本类型

**动机**

开发初期用简单的数据项（数字或字符串）表示简单的情况。但随着开发进行，数据项会需要"格式化""抽取区号"等特殊行为。一旦发现对某个数据的操作不仅仅局限于打印，就为它创建一个新类。这是最实用的重构手法之一。

**范例**

```js
// 重构前
orders.filter(o => "high" === o.priority || "rush" === o.priority);

// 重构后
class Priority {
  constructor(value) {
    if (value instanceof Priority) return value;
    if (Priority.legalValues().includes(value)) this._value = value;
    else throw new Error(`<${value}> is invalid for Priority`);
  }
  toString() { return this._value; }
  get _index() { return Priority.legalValues().findIndex(s => s === this._value); }
  static legalValues() { return ['low', 'normal', 'high', 'rush']; }
  equals(other) { return this._index === other._index; }
  higherThan(other) { return this._index > other._index; }
  lowerThan(other) { return this._index < other._index; }
}

orders.filter(o => o.priority.higherThan(new Priority("normal")));
```

### 7.4 以查询取代临时变量

**动机**

将临时变量抽取成函数，使函数分解过程更简单，不再需要将变量作为参数传递给提炼出来的小函数。同时避免在多个函数中重复编写计算逻辑。这项手法在类中施展效果最好，因为类为提炼函数提供了共同的上下文。只适用于那些只被计算一次且之后不再被修改的临时变量。

**范例**

```js
// 重构前
class Order {
  get price() {
    const basePrice = this._quantity * this._item.price;
    var discountFactor = 0.98;
    if (basePrice > 1000) discountFactor -= 0.03;
    return basePrice * discountFactor;
  }
}

// 重构后
class Order {
  get price() {
    return this.basePrice * this.discountFactor;
  }
  get basePrice() {
    return this._quantity * this._item.price;
  }
  get discountFactor() {
    var discountFactor = 0.98;
    if (this.basePrice > 1000) discountFactor -= 0.03;
    return discountFactor;
  }
}
```

### 7.5 提炼类

**动机**

类会不断成长扩展，变得过分复杂。如果某些数据和某些函数总是一起出现，或某些数据经常同时变化甚至彼此相依，就应该将它们分离到独立的类中。另一个信号：子类化只影响类的部分特性时，也需要分解原来的类。

反向重构：**内联类**

**范例**

```js
// 重构前
class Person {
  get officeAreaCode() { return this._officeAreaCode; }
  get officeNumber() { return this._officeNumber; }
}

// 重构后
class Person {
  get officeAreaCode() { return this._telephoneNumber.areaCode; }
  get officeNumber() { return this._telephoneNumber.number; }
}
class TelephoneNumber {
  get areaCode() { return this._areaCode; }
  get number() { return this._number; }
}
```

---

### 7.6 内联类（Inline Class）

反向重构：**提炼类**

**动机**

当一个类不再承担足够责任，不再有单独存在的理由时，将其塞进另一个类中。另一种场景：手头有两个类，想重新安排职责时，先用内联合并成一个类，再用**提炼类**分离职责会更简单。

**范例**

```js
// 将 TrackingInformation 内联到 Shipment
class Shipment {
  get trackingInfo() {
    return `${this.shippingCompany}: ${this.trackingNumber}`;
  }
  get shippingCompany() { return this._shippingCompany; }
  set shippingCompany(arg) { this._shippingCompany = arg; }
  get trackingNumber() { return this._trackingNumber; }
  set trackingNumber(arg) { this._trackingNumber = arg; }
}
```

### 7.7 隐藏委托关系

反向重构：**移除中间人**

**动机**

"封装"意味着每个模块应尽可能少了解系统的其他部分。如果客户端先通过服务对象的字段得到另一个对象（受托类），然后调用后者的函数，客户就必须知晓这一层委托关系。在服务对象上放置简单的委托函数，将委托关系隐藏起来，可以去除这种依赖。

**范例**

```js
// 重构前
manager = aPerson.department.manager;

// 重构后
class Person {
  get manager() { return this._department.manager; }
}
manager = aPerson.manager;
```

---

### 7.8 移除中间人

反向重构：**隐藏委托关系**

**动机**

隐藏委托关系是有代价的——每当客户端要使用受托类的新特性时，就必须在服务端添加委托函数。随着受托类的特性增多，转发函数会使人烦躁，服务类完全变成了一个中间人。此时应该让客户直接调用受托类。

**范例**

```js
// 重构前
manager = aPerson.manager;
class Person {
  get manager() { return this._department.manager; }
}

// 重构后
manager = aPerson.department.manager;
```

### 7.9 替换算法

**动机**

如果发现做一件事可以有更清晰的方式，就用比较清晰的方式取代复杂的方式。随着对问题有了更多理解，往往会发现更简单的解决方案。或者当程序库提供的功能与自己的代码重复时，也需要改变原先的算法。

**范例**

```js
// 重构前
function foundPerson(people) {
  for(let i = 0; i < people.length; i++) {
    if (people[i] === "Don") return "Don";
    if (people[i] === "John") return "John";
    if (people[i] === "Kent") return "Kent";
  }
  return "";
}

// 重构后
function foundPerson(people) {
  const candidates = ["Don", "John", "Kent"];
  return people.find(p => candidates.includes(p)) || '';
}
```

## 第 8 章 搬移特性

搬移特性是在不同的上下文之间搬移元素，使互相关联的软件要素集中到一块，让模块之间的界限更清晰。

### 8.1 搬移函数

**动机**

模块化是优秀软件设计的核心所在。任何函数都需要具备上下文环境才能存活（类、嵌套函数等模块机制）。搬移函数最直接的动因是：它频繁引用其他上下文中的元素，而对自身上下文中的元素却关心甚少。如果发现需要频繁调用一个别处的函数，也应考虑搬移它。如果需要为一整组函数创建新的上下文，可结合**函数组合成类**或**提炼类**。

**范例：搬移内嵌函数至顶层**

```js
// 重构前 - calculateDistance 嵌套在 trackSummary 中
function trackSummary(points) {
  const totalTime = calculateTime();
  const totalDistance = calculateDistance();
  const pace = totalTime / 60 / totalDistance ;
  return { time: totalTime, distance: totalDistance, pace: pace };

  function calculateDistance() {
    let result = 0;
    for (let i = 1; i < points.length; i++) {
      result += distance(points[i-1], points[i]);
    }
    return result;
  }
  function distance(p1, p2) { ... }
  function radians(degrees) { ... }
  function calculateTime() { ... }
}

// 重构后 - 搬移到顶层
function totalDistance(points) {
  let result = 0;
  for (let i = 1; i < points.length; i++) {
    result += distance(points[i-1], points[i]);
  }
  return result;
}
function distance(p1, p2) { ... }
function radians(degrees) { ... }

function trackSummary(points) {
  const totalTime = calculateTime();
  const pace = totalTime / 60 / totalDistance(points);
  return { time: totalTime, distance: totalDistance(points), pace: pace };
}
```

**范例：在类之间搬移函数**

```js
// 重构前
class Account {
  get bankCharge() {
    let result = 4.5;
    if (this._daysOverdrawn > 0) result += this.overdraftCharge;
    return result;
  }
  get overdraftCharge() {
    if (this.type.isPremium) {
      const baseCharge = 10;
      if (this.daysOverdrawn <= 7) return baseCharge;
      else return baseCharge + (this.daysOverdrawn - 7) * 0.85;
    } else {
      return this.daysOverdrawn * 1.75;
    }
  }
}
```

### 8.2 搬移字段

**动机**

数据结构是一个健壮程序的根基。如果发现某些数据放错了位置（如修改一条记录时总需要同时改动另一条记录），就应该搬移它。总是一同作为函数参数传递的数据，也应该规整到同一条记录中。如果数据已用类封装（通过访问函数存取），搬移字段会更容易。

**范例：搬移字段到共享对象**

```js
// 重构前 - 每个 Account 自己维护利率
class Account {
  constructor(number, type, interestRate) {
    this._number = number;
    this._type = type;
    this._interestRate = interestRate;
  }
  get interestRate() { return this._interestRate; }
}
class AccountType {
  constructor(nameString) { this._name = nameString; }
}

// 重构后 - 利率搬移到 AccountType（同类型账户共享利率）
class AccountType {
  constructor(nameString, interestRate) {
    this._name = nameString;
    this._interestRate = interestRate;
  }
  get interestRate() { return this._interestRate; }
}
class Account {
  constructor(number, type) {
    this._number = number;
    this._type = type;
  }
  get interestRate() { return this._type.interestRate; }
}
```

### 8.3 搬移语句到函数

反向重构：**搬移语句到调用者**

**动机**

"消除重复"是维护代码健康最重要的守则之一。如果调用某个函数时总有相同的代码需要每次执行，就应该将这段代码合并到函数中。如果某些语句与一个函数放在一起更像一个整体、更有助于理解，也应将其搬移到函数里去。

**范例**

```js
// 重构前 - 每次调用 emitPhotoData 前都有一行重复的 title 代码
function renderPerson(outStream, person) {
  result.push(`<p>title: ${person.photo.title}</p>`);
  result.concat(emitPhotoData(person.photo));
}
function photoDiv(p) {
  return [`<p>title: ${p.title}</p>`, emitPhotoData(p)];
}
function emitPhotoData(aPhoto) {
  return [
    `<p>location: ${aPhoto.location}</p>`,
    `<p>date: ${aPhoto.date.toDateString()}</p>`,
  ];
}

// 重构后 - title 搬移进 emitPhotoData
function emitPhotoData(aPhoto) {
  return [
    `<p>title: ${aPhoto.title}</p>`,
    `<p>location: ${aPhoto.location}</p>`,
    `<p>date: ${aPhoto.date.toDateString()}</p>`,
  ];
}
```

### 8.4 搬移语句到调用者

反向重构：**搬移语句到函数**

**动机**

程序抽象的边界会随系统演进发生偏移——曾经视为整体的行为可能分化出不同的关注点。当多个地方共用的行为需要在某些调用点表现出不同行为时，就把差异化部分从函数里挪出，搬移到调用处。

**范例**

```js
// 重构前 - emitPhotoData 包含三行，但 listRecentPhotos 需要单独处理 location
function emitPhotoData(outStream, photo) {
  outStream.write(`<p>title: ${photo.title}</p>\n`);
  outStream.write(`<p>date: ${photo.date.toDateString()}</p>\n`);
  outStream.write(`<p>location: ${photo.location}</p>\n`);
}

// 重构后 - location 搬移到调用者，让调用者可自定义处理
function emitPhotoData(outStream, photo) {
  outStream.write(`<p>title: ${photo.title}</p>\n`);
  outStream.write(`<p>date: ${photo.date.toDateString()}</p>\n`);
}

function renderPerson(outStream, person) {
  emitPhotoData(outStream, person.photo);
  outStream.write(`<p>location: ${person.photo.location}</p>\n`);
}

function listRecentPhotos(outStream, photos) {
  photos.filter(p => p.date > recentDateCutoff()).forEach(p => {
    emitPhotoData(outStream, p);
    outStream.write(`<p>location: ${p.location}</p>\n`);
  });
}
```

### 8.5 以函数调用取代内联代码

**动机**

一个命名良好的函数能极好地解释代码用途。如果内联代码做的事情已有现成的函数（或库函数）可以实现，就应该用函数调用取代它，以消除重复、提升表达力。

**范例**

```js
// 重构前
let appliesToMass = false;
for (const s of states) {
  if (s === "MA") appliesToMass = true;
}

// 重构后
appliesToMass = states.includes("MA");
```

### 8.6 移动语句

**动机**

让存在关联的代码一起出现，可以使代码更容易理解。把相关代码搜集到一处，往往是提炼函数等重构的准备工作。

**范例**

```js
// 重构前 - 变量声明分散
const pricingPlan = retrievePricingPlan();
const order = retreiveOrder();
let charge;
const chargePerUnit = pricingPlan.unit;

// 重构后 - 相关变量聚集
const pricingPlan = retrievePricingPlan();
const chargePerUnit = pricingPlan.unit;
const order = retreiveOrder();
let charge;
```

**范例：包含条件逻辑的移动**

```js
// 重构前 - 重复代码
let result;
if (availableResources.length === 0) {
  result = createResource();
  allocatedResources.push(result);
} else {
  result = availableResources.pop();
  allocatedResources.push(result);
}

// 重构后 - 将重复语句移出条件分支
let result;
if (availableResources.length === 0) {
  result = createResource();
} else {
  result = availableResources.pop();
}
allocatedResources.push(result);
```

### 8.7 拆分循环

**动机**

一次循环中做多件事情，修改时就需要同时理解所有事情。拆分循环让每个循环只做一件事，确保每次修改只需理解要修改的代码。后续还可将每个循环提炼成独立的函数。

**范例**

```js
// 重构前 - 一个循环做两件事
let youngest = people[0] ? people[0].age : Infinity;
let totalSalary = 0;
for (const p of people) {
  if (p.age < youngest) youngest = p.age;
  totalSalary += p.salary;
}

// 重构后 - 拆分为两个循环
function totalSalary() {
  return people.reduce((total, p) => total + p.salary, 0);
}
function youngestAge() {
  return Math.min(...people.map(p => p.age));
}
return `youngestAge: ${youngestAge()}, totalSalary: ${totalSalary()}`;
```

### 8.8 以管道取代循环

**动机**

集合管道（`map`、`filter`、`reduce` 等）让集合的迭代过程更具可读性，只需从头到尾阅读一遍就能弄清数据在管道中的变换过程。

**范例**

```js
// 重构前 - 循环处理 CSV 数据
function acquireData(input) {
  const lines = input.split("\n");
  let firstLine = true;
  const result = [];
  for (const line of lines) {
    if (firstLine) { firstLine = false; continue; }
    if (line.trim() === "") continue;
    const record = line.split(",");
    if (record[1].trim() === "India") {
      result.push({ city: record[0].trim(), phone: record[2].trim() });
    }
  }
  return result;
}

// 重构后 - 集合管道
function acquireData(input) {
  const lines = input.split("\n");
  return lines
    .slice(1)
    .filter(line => line.trim() !== "")
    .map(line => line.split(","))
    .filter(fields => fields[1].trim() === "India")
    .map(fields => ({ city: fields[0].trim(), phone: fields[2].trim() }));
}
```

### 8.9 移除死代码

**动机**

无用代码不会拖慢系统速度，但会给阅读代码的人带来额外思维负担。一旦代码不再被使用，就应该立即删除它。如果日后真的需要，可以从版本控制系统中找回。不要再注释掉代码，有版本控制系统就大胆删。

**范例**

```js
// 重构前 - 无用代码
if (false) {
  doSomethingThatUsedToMatter();
}

// 重构后 - 直接移除
```

## 第 9 章 重新组织数据

数据结构在程序中扮演着重要角色。将一个值用于多个不同的用途会催生混乱，此时应用**拆分变量**将不同用途分开。有些多余的变量最好彻底消除（**以查询取代派生变量**）。引用和值的混淆经常造成问题，可用**将引用对象改为值对象**和**将值对象改为引用对象**在两者间切换。

### 9.1 拆分变量

**动机**

变量如果被赋值超过一次，就意味它承担了不止一个责任。每个变量应该只承担一个责任，同一个变量承担两件不同的事情会令代码阅读者糊涂。（循环变量和结果收集变量是例外，它们天然会被多次赋值。）

**范例**

```js
// 重构前 - acc 被赋值两次，承担两个不同的责任
function distanceTravelled(scenario, time) {
  let result;
  let acc = scenario.primaryForce / scenario.mass;
  let primaryTime = Math.min(time, scenario.delay);
  result = 0.5 * acc * primaryTime * primaryTime;
  let secondaryTime = time - scenario.delay;
  if (secondaryTime > 0) {
    let primaryVelocity = acc * scenario.delay;
    acc = (scenario.primaryForce + scenario.secondaryForce) / scenario.mass;
    result += primaryVelocity * secondaryTime + 0.5 * acc * secondaryTime * secondaryTime;
  }
  return result;
}

// 重构后 - 拆分为 primaryAcceleration 和 secondaryAcceleration
function distanceTravelled(scenario, time) {
  let result;
  const primaryAcceleration = scenario.primaryForce / scenario.mass;
  let primaryTime = Math.min(time, scenario.delay);
  result = 0.5 * primaryAcceleration * primaryTime * primaryTime;
  let secondaryTime = time - scenario.delay;
  if (secondaryTime > 0) {
    let primaryVelocity = primaryAcceleration * scenario.delay;
    const secondaryAcceleration = (scenario.primaryForce + scenario.secondaryForce) / scenario.mass;
    result += primaryVelocity * secondaryTime + 0.5 * secondaryAcceleration * secondaryTime * secondaryTime;
  }
  return result;
}
```

**范例：对输入参数赋值**

```js
// 重构前 - inputValue 既是输入也负责返回结果
function discount(inputValue, quantity) {
  if (inputValue > 50) inputValue = inputValue - 2;
  if (quantity > 100) inputValue = inputValue - 1;
  return inputValue;
}

// 重构后 - 拆分输入值和计算结果
function discount(inputValue, quantity) {
  let result = inputValue;
  if (inputValue > 50) result = result - 2;
  if (quantity > 100) result = result - 1;
  return result;
}
```

### 9.2 字段改名

**动机**

数据结构是理解程序行为的关键，字段的命名格外重要。随着对数据理解加深，要让这些理解融入程序——给字段起更好的名字。

**范例**

```js
// 重构前
const organization = { name: "Acme Gooseberries", country: "GB" };

// 1. 先用封装记录封装成类
class Organization {
  constructor(data) {
    this._name = data.name;
    this._country = data.country;
  }
  get name() { return this._name; }
  set name(aString) { this._name = aString; }
  get country() { return this._country; }
  set country(aCountryCode) { this._country = aCountryCode; }
}

// 2. 内部字段改名，构造函数同时支持新旧字段
class Organization {
  constructor(data) {
    this._title = data.title !== undefined ? data.title : data.name;
    this._country = data.country;
  }
  get name() { return this._title; }
  set name(aString) { this._title = aString; }
  // ...
}

// 3. 逐个修改调用方后，去掉对旧字段的支持
// 4. 访问函数改名
class Organization {
  constructor(data) {
    this._title = data.title;
    this._country = data.country;
  }
  get title() { return this._title; }
  set title(aString) { this._title = aString; }
  get country() { return this._country; }
  set country(aCountryCode) { this._country = aCountryCode; }
}
```

### 9.3 以查询取代派生变量

**动机**

可变数据是最大的错误源头之一。如果某个变量可以随时计算出来，就应该去掉它——计算常能更清晰表达数据含义，也避免了"源数据修改时忘了更新派生变量"的错误。

**范例**

```js
// 重构前 - production 是手动累加的派生变量
class ProductionPlan {
  get production() { return this._production; }
  applyAdjustment(anAdjustment) {
    this._adjustments.push(anAdjustment);
    this._production += anAdjustment.amount;
  }
}

// 重构后 - production 改为即时计算
class ProductionPlan {
  get production() {
    return this._adjustments.reduce((sum, a) => sum + a.amount, 0);
  }
  applyAdjustment(anAdjustment) {
    this._adjustments.push(anAdjustment);
  }
}
```

**范例：不止一个数据来源**

```js
// 重构前 - production 有初始值 + 调整值两个来源
class ProductionPlan {
  constructor(production) {
    this._production = production;
    this._adjustments = [];
  }
  // ...
}

// 重构策略：用拆分变量将两个来源分开，然后用查询取代派生变量
class ProductionPlan {
  constructor(production) {
    this._initialProduction = production;
    this._adjustments = [];
  }
  get production() {
    return this._initialProduction + this._adjustments.reduce((sum, a) => sum + a.amount, 0);
  }
}
```

### 9.4 将引用对象改为值对象

反向重构：**将值对象改为引用对象**

**动机**

值对象通常更容易理解，主要因为它们是不可变的。不可变的数据处理起来更安全：可以放心地传给程序其他部分，不必担心被偷偷修改；可以在各处复制，不必操心内存链接。在分布式系统和并发系统中尤为有用。如果需要在几个对象之间共享一个对象并看到对共享对象的修改，则需要将其保留为引用。

**范例**

```js
// 重构前 - TelephoneNumber 是引用对象（可修改）
class Person {
  set officeAreaCode(arg) { this._telephoneNumber.areaCode = arg; }
  set officeNumber(arg) { this._telephoneNumber.number = arg; }
}
class TelephoneNumber {
  get areaCode() { return this._areaCode; }
  set areaCode(arg) { this._areaCode = arg; }
  get number() { return this._number; }
  set number(arg) { this._number = arg; }
}

// 重构后 - TelephoneNumber 变成不可变的值对象
class Person {
  set officeAreaCode(arg) {
    this._telephoneNumber = new TelephoneNumber(arg, this.officeNumber);
  }
  set officeNumber(arg) {
    this._telephoneNumber = new TelephoneNumber(this.officeAreaCode, arg);
  }
}
class TelephoneNumber {
  constructor(areaCode, number) {
    this._areaCode = areaCode;
    this._number = number;
  }
  equals(other) {
    if (!(other instanceof TelephoneNumber)) return false;
    return this.areaCode === other.areaCode && this.number === other.number;
  }
}
```

### 9.5 将值对象改为引用对象

反向重构：**将引用对象改为值对象**

**动机**

如果共享的数据需要更新，将数据复制多份就会遇到巨大困难——必须找到所有副本并全部更新。漏掉一个就会导致数据不一致。此时应将多份数据副本变成单一引用，对数据的修改会立即反映在所有关联对象中。这通常需要引入仓库（Repository）来确保每个实体只有一个对象。

**范例**

```js
// 重构前 - 每条订单创建独立的 Customer 对象（值对象）
class Order {
  constructor(data) {
    this._customer = new Customer(data.customer);
  }
}

// 添加仓库 - 确保同一个顾客 ID 只创建一个 Customer 对象
let _repositoryData = { customers: new Map() };

function registerCustomer(id) {
  if (!_repositoryData.customers.has(id))
    _repositoryData.customers.set(id, new Customer(id));
  return findCustomer(id);
}
function findCustomer(id) {
  return _repositoryData.customers.get(id);
}

// 重构后 - 从仓库获取 Customer 对象（引用对象）
class Order {
  constructor(data) {
    this._customer = registerCustomer(data.customer);
  }
}
```

## 第 10 章 简化条件逻辑

程序的大部分威力来自条件逻辑，但很不幸，程序的复杂度也大多来自条件逻辑。

### 10.1 分解条件表达式

**动机**

复杂的条件逻辑是导致复杂度上升的主要原因之一。将条件判断和每个分支分别提炼成独立函数，可以突出条件逻辑，更清楚表明每个分支的作用和原因。

**范例**

```js
// 重构前
if (!aDate.isBefore(plan.summerStart) && !aDate.isAfter(plan.summerEnd))
  charge = quantity * plan.summerRate;
else
  charge = quantity * plan.regularRate + plan.regularServiceCharge;

// 重构后
charge = summer() ? summerCharge() : regularCharge();

function summer() {
  return !aDate.isBefore(plan.summerStart) && !aDate.isAfter(plan.summerEnd);
}
function summerCharge() {
  return quantity * plan.summerRate;
}
function regularCharge() {
  return quantity * plan.regularRate + plan.regularServiceCharge;
}
```

---

### 10.2 合并条件表达式

**动机**

如果有一串条件检查，检查条件各不相同但最终行为一致，就应该用逻辑或（`||`）和逻辑与（`&&`）将它们合并。合并后可以用**提炼函数**将其提炼为独立函数，把"做什么"换成"为什么这样做"。

**范例**

```js
// 重构前
function disabilityAmount(anEmployee) {
  if (anEmployee.seniority < 2) return 0;
  if (anEmployee.monthsDisabled > 12) return 0;
  if (anEmployee.isPartTime) return 0;
  // compute the disability amount
}

// 重构后
function disabilityAmount(anEmployee) {
  if (isNotEligibleForDisability()) return 0;
  // compute the disability amount
}
function isNotEligibleForDisability() {
  return (
    anEmployee.seniority < 2 ||
    anEmployee.monthsDisabled > 12 ||
    anEmployee.isPartTime
  );
}
```

**范例：使用逻辑与**

```js
// 重构前 - 嵌套 if
if (anEmployee.onVacation)
  if (anEmployee.seniority > 10) return 1;
return 0.5;

// 重构后
if (anEmployee.onVacation && anEmployee.seniority > 10) return 1;
return 0.5;
```

---

### 10.3 以卫语句取代嵌套条件表达式

**动机**

条件表达式有两种风格：两个分支都是正常行为（用 `if...else`），或只有一个分支是正常行为，另一个是异常情况（用卫语句）。卫语句告诉阅读者："这种情况不是本函数的核心逻辑，如果发生就退出。"不要迷信"每个函数只能有一个出口"，保持代码清晰才最关键。

**范例**

```js
// 重构前 - 嵌套条件
function getPayAmount() {
  let result;
  if (isDead) result = deadAmount();
  else {
    if (isSeparated) result = separatedAmount();
    else {
      if (isRetired) result = retiredAmount();
      else result = normalPayAmount();
    }
  }
  return result;
}

// 重构后 - 卫语句
function getPayAmount() {
  if (isDead) return deadAmount();
  if (isSeparated) return separatedAmount();
  if (isRetired) return retiredAmount();
  return normalPayAmount();
}
```

**范例：将条件反转**

```js
// 重构前
function adjustedCapital(anInstrument) {
  let result = 0;
  if (anInstrument.capital > 0) {
    if (anInstrument.interestRate > 0 && anInstrument.duration > 0) {
      result = (anInstrument.income / anInstrument.duration) * anInstrument.adjustmentFactor;
    }
  }
  return result;
}

// 重构后 - 反转条件 + 合并 + 移除多余变量
function adjustedCapital(anInstrument) {
  if (anInstrument.capital <= 0 || anInstrument.interestRate <= 0 || anInstrument.duration <= 0) return 0;
  return (anInstrument.income / anInstrument.duration) * anInstrument.adjustmentFactor;
}
```

---

### 10.4 以多态取代条件表达式

**动机**

复杂的条件逻辑可以拆分为不同的场景。多态能让拆分表述得更清晰。常见情况：多个函数都有基于类型代码的 switch 语句（为每种分支创建子类）；或基础逻辑 + 变体逻辑（基础逻辑放进超类，变体放进子类）。但不是所有条件逻辑都需要多态，简单 if/else 不必劳师动众。

**范例**

```js
// 重构前 - switch 分散在多个函数中
function plumage(bird) {
  switch (bird.type) {
    case 'EuropeanSwallow': return "average";
    case 'AfricanSwallow':  return (bird.numberOfCoconuts > 2) ? "tired" : "average";
    case 'NorwegianBlueParrot': return (bird.voltage > 100) ? "scorched" : "beautiful";
    default: return "unknown";
  }
}
function airSpeedVelocity(bird) {
  switch (bird.type) {
    case 'EuropeanSwallow': return 35;
    case 'AfricanSwallow':  return 40 - 2 * bird.numberOfCoconuts;
    case 'NorwegianBlueParrot': return (bird.isNailed) ? 0 : 10 + bird.voltage / 10;
    default: return null;
  }
}

// 重构后 - 多态
function createBird(bird) {
  switch (bird.type) {
    case 'EuropeanSwallow':    return new EuropeanSwallow(bird);
    case 'AfricanSwallow':     return new AfricanSwallow(bird);
    case 'NorwegianBlueParrot':return new NorwegianBlueParrot(bird);
    default:                   return new Bird(bird);
  }
}
class Bird {
  get plumage() { return "unknown"; }
  get airSpeedVelocity() { return null; }
}
class EuropeanSwallow extends Bird {
  get plumage() { return "average"; }
  get airSpeedVelocity() { return 35; }
}
class AfricanSwallow extends Bird {
  get plumage() { return (this.numberOfCoconuts > 2) ? "tired" : "average"; }
  get airSpeedVelocity() { return 40 - 2 * this.numberOfCoconuts; }
}
class NorwegianBlueParrot extends Bird {
  get plumage() { return (this.voltage > 100) ? "scorched" : "beautiful"; }
  get airSpeedVelocity() { return (this.isNailed) ? 0 : 10 + this.voltage / 10; }
}
```

### 10.5 引入特例

曾用名：引入 Null 对象

**动机**

当多处代码以同样的方式应对同一个特殊值（如 `null` 或 `"unknown"`）时，创建一个特例对象封装这些处理逻辑。这样可以用函数调用取代大部分特例检查逻辑。Null 对象是特例的一种特例。

**范例**

```js
// 重构前 - 处处检查 "unknown"
const customerName = aCustomer === "unknown" ? "occupant" : aCustomer.name;
const plan = aCustomer === "unknown" ? registry.billingPlans.basic : aCustomer.billingPlan;

// 重构后 - 使用特例对象
class UnknownCustomer {
  get isUnknown() { return true; }
  get name() { return "occupant"; }
  get billingPlan() { return registry.billingPlans.basic; }
  set billingPlan(arg) { /* ignore */ }
  get paymentHistory() { return new NullPaymentHistory(); }
}
class NullPaymentHistory {
  get weeksDelinquentInLastYear() { return 0; }
}

// 客户端不再需要特例检查
const customerName = aCustomer.name;
const plan = aCustomer.billingPlan;
const weeksDelinquent = aCustomer.paymentHistory.weeksDelinquentInLastYear;
```

**范例：使用对象字面量（只读数据）**

```js
function createUnknownCustomer() {
  return Object.freeze({
    isUnknown: true,
    name: "occupant",
    billingPlan: registry.billingPlans.basic,
    paymentHistory: { weeksDelinquentInLastYear: 0 },
  });
}
```

### 10.6 引入断言

**动机**

当一段代码假设某个条件始终为真时，用断言明确标明这个假设。断言是一种有价值的交流形式，告诉阅读者程序在执行到此处时的状态假设。断言对调试也很有帮助，且应该永久保留。注意不要滥用——只检查"必须为真"的条件，而非"应该为真"的条件。

**范例**

```js
// 重构前 - 假设 discountRate 合法
applyDiscount(aNumber) {
  return (this.discountRate)
    ? aNumber - (this.discountRate * aNumber)
    : aNumber;
}

// 重构后 - 在设值函数上添加断言
set discountRate(aNumber) {
  assert(null === aNumber || aNumber >= 0);
  this._discountRate = aNumber;
}
```

## 第 11 章 重构 API

模块和函数是软件的骨肉，而 API 则是将骨肉连接起来的关节。

### 11.1 将查询函数和修改函数分离

**动机**

任何有返回值的函数，都不应该有看得到的副作用（命令与查询分离原则）。明确区分有副作用和无副作用的函数，能提升代码的可理解性和可测试性。

**范例**

```js
// 重构前 - 既有返回值又有副作用的函数
function alertForMiscreant(people) {
  for (const p of people) {
    if (p === "Don") { setOffAlarms(); return "Don"; }
    if (p === "John") { setOffAlarms(); return "John"; }
  }
  return "";
}

// 重构后 - 分离为查询函数和修改函数
function findMiscreant(people) {
  for (const p of people) {
    if (p === "Don") return "Don";
    if (p === "John") return "John";
  }
  return "";
}
function alertForMiscreant(people) {
  if (findMiscreant(people) !== "") setOffAlarms();
}
```

---

### 11.2 函数参数化

**动机**

如果两个函数逻辑非常相似，只有一些字面量值不同，可以合并成一个函数，以参数的形式传入不同的值，从而消除重复。

**范例**

```js
// 重构前
function tenPercentRaise(aPerson) {
  aPerson.salary = aPerson.salary.multiply(1.1);
}
function fivePercentRaise(aPerson) {
  aPerson.salary = aPerson.salary.multiply(1.05);
}

// 重构后
function raise(aPerson, factor) {
  aPerson.salary = aPerson.salary.multiply(1 + factor);
}
```

---

### 11.3 移除标记参数

**动机**

标记参数（如布尔值 `true`/`false` 或枚举）隐藏了函数调用中的差异性，让人难以理解到底有哪些函数可以调用、应该怎么调用。布尔型的标记尤其糟糕，因为调用时很难弄清 `true` 到底是什么意思。用明确函数替代标记参数，能更清晰地表达调用者的意图。

**范例**

```js
// 重构前 - 通过布尔标记区分两种行为
aShipment.deliveryDate = deliveryDate(anOrder, true);
aShipment.deliveryDate = deliveryDate(anOrder, false);

// 重构后 - 两个明确函数
aShipment.deliveryDate = rushDeliveryDate(anOrder);
aShipment.deliveryDate = regularDeliveryDate(anOrder);
```

---

### 11.4 保持对象完整

**动机**

如果代码从一个记录结构中导出几个值，再一起传递给函数，不如直接传递整个记录。这样可以更好地应对变化（被调函数需要更多数据时无需修改参数列表），也能缩短参数列表，让函数调用更容易看懂。

**范例**

```js
// 重构前 - 拆开传递
const low = aRoom.daysTempRange.low;
const high = aRoom.daysTempRange.high;
if (!aPlan.withinRange(low, high))
  alerts.push("room temperature went outside range");

// 重构后 - 传递完整对象
if (!aPlan.withinRange(aRoom.daysTempRange))
  alerts.push("room temperature went outside range");
```

---

### 11.5 以查询取代参数

反向重构：**以参数取代查询**

**动机**

函数的参数列表应该总结该函数的可变性。如果函数自己可以轻松获得某个参数的值，这就是重复，会增加调用者的负担。移除参数后，获取值的责任从调用者转移到函数本身。

**范例**

```js
// 重构前
class Order {
  get finalPrice() {
    const basePrice = this.quantity * this.itemPrice;
    return this.discountedPrice(basePrice, this.discountLevel);
  }
  get discountLevel() {
    return (this.quantity > 100) ? 2 : 1;
  }
  discountedPrice(basePrice, discountLevel) {
    switch (discountLevel) {
      case 1: return basePrice * 0.95;
      case 2: return basePrice * 0.9;
    }
  }
}

// 重构后 - discountedPrice 自己调用 discountLevel，无需传入
class Order {
  get finalPrice() {
    const basePrice = this.quantity * this.itemPrice;
    return this.discountedPrice(basePrice);
  }
  discountedPrice(basePrice) {
    switch (this.discountLevel) {
      case 1: return basePrice * 0.95;
      case 2: return basePrice * 0.9;
    }
  }
}
```

### 11.6 以参数取代查询

反向重构：**以查询取代参数**

**动机**

有时函数体内部引用了不想要的全局变量或其他元素。将查询结果改为参数传入，可以打破函数的依赖关系，使其具备引用透明性（同样的参数总是返回同样的结果），更易测试和理解。副作用是将处理依赖的责任转移给了调用者。

**范例**

```js
// 重构前 - 依赖于全局 thermostat 对象
class HeatingPlan {
  get targetTemperature() {
    if (thermostat.selectedTemperature > this._max) return this._max;
    else if (thermostat.selectedTemperature < this._min) return this._min;
    else return thermostat.selectedTemperature;
  }
}

// 重构后 - 将依赖作为参数传入
class HeatingPlan {
  targetTemperature(selectedTemperature) {
    if (selectedTemperature > this._max) return this._max;
    else if (selectedTemperature < this._min) return this._min;
    else return selectedTemperature;
  }
}
```

### 11.7 移除设值函数

**动机**

如果字段在对象创建后不应该再被修改，就不应为它提供设值函数。这能使"不想让它被修改"的意图更清晰，并排除其值被意外修改的可能性。

**范例**

```js
// 重构前 - id 有设值函数
class Person {
  get name() { return this._name; }
  set name(arg) { this._name = arg; }
  get id() { return this._id; }
  set id(arg) { this._id = arg; }
}
const martin = new Person();
martin.name = "martin";
martin.id = "1234";

// 重构后 - id 通过构造函数设值，移除设值函数
class Person {
  constructor(id) { this._id = id; }
  get name() { return this._name; }
  set name(arg) { this._name = arg; }
  get id() { return this._id; }
}
const martin = new Person("1234");
martin.name = "martin";
```

### 11.8 以工厂函数取代构造函数

**动机**

构造函数有许多局限性：只能返回当前类的实例、名字固定、需通过 `new` 关键字调用。工厂函数不受这些限制，可以返回子类或代理对象，也可以用更清晰的名字。

**范例**

```js
// 重构前
const leadEngineer = new Employee(document.leadEngineer, "E");

// 重构后
function createEngineer(name) {
  return new Employee(name, "E");
}
const leadEngineer = createEngineer(document.leadEngineer);
```

### 11.9 以命令取代函数

反向重构：**以函数取代命令**

**动机**

将函数封装成命令对象，可以获得更大的控制灵活性和更强的表达能力：支持撤销操作、更丰富的生命周期管理、可借助继承和钩子定制行为。适合拆解复杂函数。不过命令对象以复杂性为代价，95% 的情况下普通函数就足够了。

**范例**

```js
// 重构前 - 复杂函数
function score(candidate, medicalExam, scoringGuide) {
  let result = 0;
  let healthLevel = 0;
  // ... long body code
  return result;
}

// 重构后 - 命令对象，便于拆解
class Scorer {
  constructor(candidate, medicalExam, scoringGuide) {
    this._candidate = candidate;
    this._medicalExam = medicalExam;
    this._scoringGuide = scoringGuide;
  }
  execute() {
    this._result = 0;
    this._healthLevel = 0;
    // ... long body code
    return this._result;
  }
}

function score(candidate, medicalExam, scoringGuide) {
  return new Scorer(candidate, medicalExam, scoringGuide).execute();
}
```

### 11.10 以函数取代命令

反向重构：**以命令取代函数**

**动机**

如果命令对象足够简单，变成函数更合适——大多数时候只是为了调用一个函数完成工作，命令对象可能显得费而不惠。

**范例**

```js
// 重构前 - 命令对象过于简单
class ChargeCalculator {
  constructor(customer, usage, provider) {
    this._customer = customer;
    this._usage = usage;
    this._provider = provider;
  }
  get charge() {
    return this._customer.baseRate * this._usage + this._provider.connectionCharge;
  }
}

// 重构后 - 普通函数
function charge(customer, usage, provider) {
  const baseCharge = customer.baseRate * usage;
  return baseCharge + provider.connectionCharge;
}
```

## 第 12 章 处理继承关系

继承是一个强有力的工具，但也经常被误用。本章介绍如何在继承体系里上下调整特性，以及在继承不再合适时用委托取代继承。

### 12.1 函数上移

反向重构：**函数下移**

**动机**

避免重复代码很重要。如果各个子类中都有相同（或可通过参数化统一）的函数，就将其提升到超类中。

**范例**

```js
// 重构前
class Employee extends Party {
  get annualCost() { return this.monthlyCost * 12; }
}
class Department extends Party {
  get totalAnnualCost() { return this.monthlyCost * 12; }
}

// 重构后 - 统一函数名并上移到超类
class Party {
  get annualCost() { return this.monthlyCost * 12; }
}
class Employee extends Party {}
class Department extends Party {}
```

---

### 12.2 字段上移

反向重构：**字段下移**

**动机**

如果各子类拥有相同的字段（或被以相同方式使用的字段），就将其提升到超类中，去除重复的数据声明。

**做法**

检查字段使用方式 → 统一字段名 → 在超类中新建字段 → 移除子类中的字段。

---

### 12.3 构造函数本体上移

**动机**

构造函数有特殊的规则，不能直接通过**提炼函数**和**函数上移**来处理。子类构造函数中的公共代码应提升到超类构造函数中。

**范例**

```js
// 重构前
class Employee extends Party {
  constructor(name, id, monthlyCost) {
    super();
    this._id = id;
    this._name = name;        // 公共代码
    this._monthlyCost = monthlyCost;
  }
}
class Department extends Party {
  constructor(name, staff) {
    super();
    this._name = name;        // 公共代码
    this._staff = staff;
  }
}

// 重构后
class Party {
  constructor(name) { this._name = name; }
}
class Employee extends Party {
  constructor(name, id, monthlyCost) {
    super(name);
    this._id = id;
    this._monthlyCost = monthlyCost;
  }
}
class Department extends Party {
  constructor(name, staff) {
    super(name);
    this._staff = staff;
  }
}
```

---

### 12.4 函数下移

反向重构：**函数上移**

**动机**

如果超类中的某个函数只与一个（或少数几个）子类有关，就将其下移到真正关心它的子类中。

**做法**

将超类函数本体复制到需要此函数的子类 → 删除超类中的函数 → 从不需要的子类中删除。

---

### 12.5 字段下移

反向重构：**字段上移**

**动机**

如果某个字段只被一个（或一小部分）子类用到，就将其搬移到需要该字段的子类中。

**做法**

在需要该字段的子类中声明该字段 → 从超类中移除 → 从不需要的子类中删除。

---

### 12.6 以子类取代类型码

反向重构：**移除子类**

**动机**

如果有一个类型码字段，其取值决定了不同的行为，可以引入子类，利用多态来处理条件逻辑。子类还能让"只对特定类型有意义"的字段和函数得到更清晰的表达。

**范例**

```js
// 重构前
function createEmployee(name, type) {
  return new Employee(name, type);
}

// 重构后 - 工厂函数 + 子类
function createEmployee(name, type) {
  switch (type) {
    case "engineer": return new Engineer(name);
    case "salesman": return new Salesman(name);
    case "manager":  return new Manager(name);
    default: throw new Error(`Employee cannot be of type ${type}`);
  }
}
class Employee {
  constructor(name) { this._name = name; }
}
class Engineer extends Employee {}
class Salesman extends Employee {}
class Manager extends Employee {}
```

---

### 12.7 移除子类

反向重构：**以子类取代类型码**

**动机**

子类存在就有成本。如果子类用处太少（例如只覆写了一个取值函数），不值得独立存在，就将其替换为超类中的一个字段。

**范例**

```js
// 重构前 - 子类只覆写 genderCode
class Person {
  get genderCode() { return "X"; }
}
class Male extends Person {
  get genderCode() { return "M"; }
}
class Female extends Person {
  get genderCode() { return "F"; }
}

// 重构后 - 用字段取代子类
class Person {
  constructor(name, genderCode) {
    this._name = name;
    this._genderCode = genderCode || "X";
  }
  get genderCode() { return this._genderCode; }
}
function createPerson(aRecord) {
  switch (aRecord.gender) {
    case "M": return new Person(aRecord.name, "M");
    case "F": return new Person(aRecord.name, "F");
    default:  return new Person(aRecord.name, "X");
  }
}
```

---

### 12.8 提炼超类

**动机**

如果两个类在做相似的事，就把它们的共同行为和数据提炼到超类中。合理的继承关系常常在程序演化过程中浮现，而不是预先设计好的。

**范例**

```js
// 重构前
class Employee {
  get name() { return this._name; }
  get annualCost() { return this.monthlyCost * 12; }
}
class Department {
  get name() { return this._name; }
  get totalAnnualCost() { return this.totalMonthlyCost * 12; }
}

// 重构后
class Party {
  get name() { return this._name; }
  get annualCost() { return this.monthlyCost * 12; }
}
class Employee extends Party {}
class Department extends Party {
  get monthlyCost() { /* 计算部门月度成本 */ }
}
```

---

### 12.9 折叠继承体系

**动机**

当一个子类与其超类已经没多大差别时，就把它们合并成一个类。

**做法**

选择要移除的类 → 使用字段/函数上移/下移把所有元素集中到一个类中 → 调整所有引用点 → 移除空类。

---

### 12.10 以委托取代子类

**动机**

继承只能处理一个方向上的变化，且引入了紧密耦合。当需要动态改变对象的行为、或想用继承处理其他变化方向时，就用委托取代子类。这就是"对象组合优于类继承"的实践。

**范例**

有一个演出预订系统。普通预订和高级预订在多个行为上有差异，但"是否为高级预订"这个状态可能需要在运行中动态改变（例如用户中途升级）。

```js
// 重构前 - 通过子类处理差异
class Booking {
  constructor(show, date) {
    this._show = show;
    this._date = date;
  }
  get hasTalkback() {
    // 普通预订：只有非高峰日才提供
    return this._show.hasOwnProperty('talkback') && !this.isPeakDay;
  }
  get basePrice() {
    let result = this._show.price;
    if (this.isPeakDay) result += Math.round(result * 0.15);
    return result;
  }
  get isPeakDay() {
    return this._date > new Date('2024-12-20') && this._date < new Date('2025-01-06');
  }
}
class PremiumBooking extends Booking {
  constructor(show, date, extras) {
    super(show, date);
    this._extras = extras;
  }
  get hasTalkback() {
    return this._show.hasOwnProperty('talkback'); // 任何一天都提供
  }
  get basePrice() {
    return Math.round(super.basePrice + this._extras.premiumFee);
  }
  get hasDinner() {
    return this._extras.hasOwnProperty('dinner') && !this.isPeakDay;
  }
}

// 使用时无法动态切换
let booking = new PremiumBooking(show, date, extras);
// 无法变成：booking = 降级为普通预订（除非重建整个对象）
```

```js
// 重构后 - 用委托对象表达差异
class Booking {
  constructor(show, date) {
    this._show = show;
    this._date = date;
    this._premiumDelegate = null;
  }
  // 动态升级/降级的能力
  bePremium(extras) {
    this._premiumDelegate = new PremiumDelegate(this, extras);
  }
  beRegular() {
    this._premiumDelegate = null;
  }

  get hasTalkback() {
    return this._premiumDelegate
      ? this._premiumDelegate.hasTalkback
      : this._show.hasOwnProperty('talkback') && !this.isPeakDay;
  }
  get basePrice() {
    let result = this._show.price;
    if (this.isPeakDay) result += Math.round(result * 0.15);
    return this._premiumDelegate
      ? this._premiumDelegate.extendBasePrice(result)
      : result;
  }
  get hasDinner() {
    return this._premiumDelegate
      ? this._premiumDelegate.hasDinner
      : undefined;
  }
  get isPeakDay() {
    return this._date > new Date('2024-12-20') && this._date < new Date('2025-01-06');
  }
}

// 委托类封装高级预订特有的行为
class PremiumDelegate {
  constructor(host, extras) {
    this._host = host;      // 反向引用宿主 Booking 对象
    this._extras = extras;  // 高级预订的附加服务
  }
  get hasTalkback() {
    return this._host._show.hasOwnProperty('talkback');
  }
  extendBasePrice(base) {
    return Math.round(base + this._extras.premiumFee);
  }
  get hasDinner() {
    return this._extras.hasOwnProperty('dinner') && !this._host.isPeakDay;
  }
}

// 使用时可以动态切换
let booking = new Booking(show, date);
booking.bePremium(extras);   // 升级为高级预订
booking.beRegular();         // 降级回普通预订
```

### 12.11 以委托取代超类

**动机**

当超类的一些函数对子类并不适用时，说明不应该通过继承来复用功能。合理的继承应该满足"子类的所有实例都是超类的实例"。如果子类与超类耦合过强，也应考虑用委托取代超类。

**范例**

以"栈"继承"列表"的经典错误为例：栈只需要 `push`、`pop`、`peek` 操作，但继承列表后却暴露了 `insertAt`、`removeAt`、`sort` 等不适用于栈的方法，破坏了栈的封装性。

```js
// 重构前 - Stack 继承 List，暴露了不该有的方法
class List {
  constructor() { this._items = []; }
  get(index) { return this._items[index]; }
  add(item) { this._items.push(item); }
  removeAt(index) { this._items.splice(index, 1); }
  insertAt(item, index) { this._items.splice(index, 0, item); }
  size() { return this._items.length; }
  sort() { this._items.sort(); }
}

class Stack extends List {
  push(item) { this.add(item); }
  pop() { return this._items.pop(); }
  peek() { return this._items[this._items.length - 1]; }
}

// 问题：外部可以调用栈不应支持的操作
const stack = new Stack();
stack.push(1);            // ✅ 栈的正确用法
stack.sort();             // ❌ 栈不应该能排序！
stack.insertAt(5, 0);     // ❌ 栈不应该能中间插入！
stack instanceof List;    // true，但栈不是列表
```

```js
// 重构后 - Stack 委托给 List，只暴露需要的接口
class Stack {
  constructor() {
    this._storage = new List();  // 委托，而非继承
  }
  push(item) { this._storage.add(item); }
  pop() { return this._storage._items.pop(); }
  peek() { return this._storage._items[this._storage.size() - 1]; }
  size() { return this._storage.size(); }
}

// 使用：只能操作栈允许的方法
const stack = new Stack();
stack.push(1);
stack.pop();              // ✅ 可以
stack.sort();             // ❌ 编译/运行时错误，不存在此方法
stack instanceof List;    // false，栈不是列表，语义正确
```
